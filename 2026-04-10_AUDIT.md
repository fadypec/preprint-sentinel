## Critical Issues

### 1. Secrets committed to git history

**Severity:** CRITICAL
**Location:** `.env` file in repository

The `.env` file containing the Anthropic API key, NCBI API key, and email addresses has been committed to git. Even if removed now, credentials remain in git history.

**Required action:**
1. Rotate all API keys immediately (Anthropic dashboard, NCBI account)
2. Remove `.env` from tracking: `git rm --cached .env`
3. Verify `.env` is in `.gitignore`
4. Consider `git filter-repo` to purge from history if repo is public

### 2. Server actions missing authorisation checks

**Severity:** CRITICAL
**Location:** `dashboard/app/pipeline/actions.ts`

`triggerPipeline()`, `cancelPipeline()`, `clearRunHistory()`, and `togglePubmedQueryMode()` are server actions without `requireAdmin()` guards. Any authenticated user (analyst role) could trigger or cancel pipeline runs.

**Fix:** Add `await requireAdmin()` at the start of each server action.

### 3. No health check endpoint

**Severity:** CRITICAL

No endpoint exists to verify system health: database connectivity, scheduler state, time since last successful pipeline run, API key validity. The system can be silently broken for days.

**Fix:** Add `GET /api/health` returning structured status (database, scheduler, last run timestamp, hours since run).

### 4. Pipeline failure alerting incomplete

**Severity:** CRITICAL
**Location:** `scripts/monitor_data_corruption.py:219`

Pipeline failures are logged but not alerted. The corruption monitor has a placeholder for email alerting (`"placeholder for email alerting"`). A failed run may not be discovered until the next manual check.

**Fix:** Wire pipeline failure events to the existing Slack/email alert infrastructure.

## Medium Severity Issues

### Efficiency & Performance

- `PaperCard` component rendered in list without `React.memo` — unnecessary re-renders on filter changes
- Recharts library imported on main bundle — should be dynamically imported only on analytics page
- `parseDimensions()` runs client-side on every render — should be server-side

### Security

- Rate limiting is in-memory only — documented limitation, acceptable for single-server
- Rate limiting is per-IP not per-user — spoofable via `X-Forwarded-For`
- Settings secrets (Slack webhook) stored in database JSONB — should use environment variables or secrets manager

### Code Quality

- No retry on transient database errors (connection drops fail entire stage)
- Broad `except Exception` in ingest loop masks programming bugs alongside API failures
- Missing input validation on `config.py` — invalid threshold values (e.g., `coarse_filter_threshold=2.5`) silently accepted
- `methods_analysis.py` batch API: failed individual items not re-submitted to sync API

### Operations

- Dashboard Vitest suite (`npm test`) not executed in CI — only lint and type-check run
- No coverage threshold enforcement (`--cov-fail-under` not set)
- No operations runbook for emergencies (DB restore, API key rotation, stuck pipeline)
- No backup verification — backups created but never test-restored
- No dependency vulnerability scanning (Dependabot/Snyk not configured)

### Accessibility

- Focus ring opacity at 50% — may not be visible enough for keyboard users
- Disabled pagination uses `<span>` with `pointer-events-none` — not keyboard accessible
- Dynamic content changes (analyst notes save) should use `role="status"` alongside `aria-live`
